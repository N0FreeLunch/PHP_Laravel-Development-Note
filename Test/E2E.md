# 엔드 투 엔드 테스트

## E2E 테스트의 단점

대규모의 E2E 테스트는 느리기 때문에 매번 테스트 할 때 많은 시간이 든다.

사람이 직접 돔에 접근하는 코드를 작성하는 경우 코드 작성하는 시간이 많이 필요하다.

## E2E 테스트의 장점

어플리케이션의 실제 동작을 한 동작 단위로 확인을 할 수 있다.

브라우저상에서의 실제 동작을 테스트 할 수 있기 때문에 직관적으로 테스트하기 쉽다.

개발자 뿐만 아니라, 실제 프로덕트를 운용하는 사람들도 테스트 결과 및 동작을 확인하기 쉽다.

## AI 시대 다시 E2E 테스트

요즘 AI가 너무 좋아져서 AI가 작성한 모든 코드 다 읽을 필요가 없다고 느낀다. 특히 에이전트 환경에 각종 가이드 라인이 잘 세팅된 경우 사양대로 코드가 잘 작성될 확률은 무척 높다.

코드를 작성하는 것도 AI, 테스트를 작성하는 것도 AI. 남이 짠 코드이기 때문에 이해하는데 시간이 들고 AI가 짠 코드를 읽고 해석하는 것은 병목이 된다.

이런 조건/분기에는 화면에 이런 결과가 나와야 하고, 저런 조건/분기에는 화면에 저런 결과가 나와야 한다는 것을 사람이 코드를 확인하는 것은, 사양서 확인과 코드 확인의 2중의 과정이고, 머릿속으로 AI가 짠 코드의 조건/분기를 추적하여 결과가 맞는지를 생각하는 것은 피곤하고 시간이 들어가는 일이다.

코드 확인의 시간은 정말 투자해야 하는 시간인가? 더 나은 생산성을 위해 비중을 줄여야 하는 부분인가? 고민을 하는 시기이다.

E2E 테스트는 웹 화면을 보고 동작을 확인할 수 있다. 코드를 보는 것 보다, 동작을 보고 확인하는 것이 간단하고 이해하기 쉽다.

내가 정한 사양대로, 결과가 잘 동작하는지 확인하면, 코드의 로직을 모두 확인할 필요가 없는데, 이는 잘 정의된 사양이라면 AI는 대부분의 코드를 정확하게 작성할 수 있다는 것을 요즘 체감하고 있기 때문에 코드를 해석하는 비중을 줄이는 것이 낫지 않을까 하는 판단이 들기 때문이다.

## AI로 E2E 테스트 작성하기

E2E 테스트 코드를 짤 때 AI는 일반 코드를 작성하는 것 보다 더 많은 토큰을 소모한다.

하나의 기능을 작성할 때, 사양 + 코드 + 테스트의 코드를 작성하는데, 시나리오 + E2E 테스트의 과정이 별도로 추가가 되는 것이다.

그리고 E2E 테스트를 작성할 때, 브라우저의 결과를 해석하고 이에 맞는 동작을 정의하는 과정에서 프로젝트의 각종 코드와 제약 사항을 확인하는데 이 때 막대한 토큰이 소모된다.

한 두개의 시나리오를 AI로 E2E 테스트를 작성하는데, 26년 4월 중순 기준 비즈니스 플렌 또는 일반 플렌으로 5시간 리밋이 1시간도 안 되어 소모되고, 주간 사용량의 15% 정도를 사용한다.

사람이 데이터 조건을 세팅하고 직접 화면의 여러 동작을 확인하는 것 보다 사양을 정의할 때 이미 화면이 어떻게 전이 되어야 한다는 것을 알고 있기 때문에 간략한 동작 흐름만 알려 주면, AI가 알아서 버튼을 찾고 클릭을 하는 모든 과정이 그려진다.

코드를 확인하고, 이 조건일 때 이 동작이 일어난다는 과정을 모두 추적할 필요가 없어진다. 정말 이건 확인해 보는 게 좋겠다는 것 몇 부분만 찝거나, 이상해 보이는 코드가 없는지 확인하는 정도로 충분할 수 있다.

## 시나리오

유저 또는 운용자가 화면을 움직이고, 동작을 확인하는 과정으로, 개발자는 하나의 항목 변경이 프로젝트의 여러 화면에 어떤 영향을 끼치는지 알 수 있다. 이를 통해서 해당 항목 변경으로 유도되는 모든 변경 지점을 테스트 할 수 있다.
