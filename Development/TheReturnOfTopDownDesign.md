# 탑 다운 디자인의 귀환

## 왜 탑 다운인가?

탑 다운은 사양을 의뢰하고 발주하기 위해서 사용한다. 소프트웨어 제작에 필요한 대략적인 비용을 산출하고 일정을 정하기 위해서 어쩔 수 없이 선택하는 방법이었다.

탑다운을 잘 할 수 있도록 전체를 설계를 잘 할 수 있는 사람은 드물었고, 탑 다운 방식에 대한 고객의 수요는 있었기 때문에 상세 설계가 잘 이뤄지지 않는 경우가 많았고, 잘못된 설계와 코드를 억지로 전체 사양에 끼워 맞추는 바람에 지속적인 메인테넌스가 곤란하게 되었다.

한국에서 새 제품을 만들 때는 SI 개발자로, 개발된 제품을 메인테넌스 하는 SM 개발자로 나누는 이유도, SI는 빠르게 제품을 개발해서 돈을 많이 벌고, 엉망이 코드들 메인테넌스 하는 비용은 많이 들기 때문에 비교적 몸값이 적은 개발자들을 SM 개발에 맡기기 위함이다.

## 왜 애자일인가?

많은 기업에서 애자일을 채택하는 이유는 일단 보여줄 수 있는 단계까지의 기능을 빠르게 만들고, 짧은 단위로 개발 성과를 어필하기 위함이다.

작은 단위로 만든 기능을 계속 보여주면서 어느 정도 만들어진 단계에서 다음 단계를 생각할 수 있기 때문에 좀 더 탄탄하게 비즈니스 로직을 쌓아갈 수 있다는 장점이 있었다.

하지만 전체적인 개발의 공수 보다는 티켓의 개발 공수 정도를 산정해야 하는 문제가 있었고, 티켓마다 일정이 저마다 다른다는 문제점이 있어 거대한 규모의 개발 시간과 공수를 예측하기는 어려웠다.

## AI 시대

AI가 어느 정도 훌륭하게 코드를 작성해 주는 시대가 되었다. 하지만 AI가 비즈니스에서 원하는 코드를 작성해 주지 않았고, 비즈니스에서 원하는 사양을 정하다보면 차라리 코드를 작성하는 것이 시간이 빠른 경우도 있어서 AI는 보조 도구로 활용되었다.

AI의 코드 작성 능력을 최대한 활용을 해서 생산성을 늘리기 위해서는 AI가 이해하고 판단하기 쉬운 환경을 만들고 코드를 작성하게 하는 전략이 필요하게 되었다.

비즈니스 로직과 코드가 거의 일치하는 개발, 그리고 잘 모듈화 되어 기능이나 도메인의 단위가 분리가 잘 되어 있어서 AI가 해석하기 좋은 코드가 되어야 했다.

생산성 때문에 개발자는 보일러 플레이트가 적은 언어를 많이 채택했지만, AI에게 보일러플레이트는 특별한 문제가 되지 않았고 이 때문에 도메인 주도 설계 아키텍처 설계 능력이 AI를 더 잘 활용할 수 있는 수단이 되었다.
