# Ai를 통한 개발 방법론

## 하네스 엔지니어링

### 하네스 엔지니어링이란?

- 확률론적인 ai의 코드를 어떻게 정교하게 통제 할 것인가
- 인간이 모든 코드를 다 이해할 수는 없다.
- AI의 잘못된 코드를 최대한 올바르게 만들어야 한다는 화두에서 출발

### 하네스 엔지니어링 방법

- 타입 사용하기, 정적 분석 사용하기, 정적 분석을 만족하는 코드를 작성할 수 있도록 에이전트 스킬 및 에이전트 md 파일을 세팅하기 (세팅을 하지 않으면 정적 분석을 만족하지 않는 코드를 작성해서 여러번 수정을 해야 한다.)
- phpstan 최고 단계 Lv.9를 목표로 하기 (예전에는 LV5 정도 수준을 지켜야 했으나, 사람이 코드를 안 짜니까 AI의 코드 생성을 최대한 활용한다.)
- assert와 phpdoc 타입을 적극적으로 이용한다 (개인적으로는 phpstan 레퍼런스 문서를 적극적으로 공부하는 것을 추천, AI에 질문하면서 공부할 것)

## 코드를 늘려라

- 옛날에는 코드가 적은 것이 좋았다. 관리할 포인트가 줄어들기 때문이다.

### 타입을 적극 이용하자

- AI 시대 코드 생성 비용이 저렴해졌다. 이전까지는 시간이 걸려서 생략한, DTO, VO, Enum 등을 최대한 많이 추가하여 타입 안전하게 만들도록 하자.
- AGENTS.md, AGENT SKILL 등에 DTO, VO, Enum 등을 사용해야 한다는 지침을 만들자.
- VO는 반드시 유닛 테스트를 생성하자 (실수한 적이 여러번 있다.)
- 타입을 통해서 AI의 실수를 좀 더 줄일 수 있다.

### AI를 위한 예외 메시지 만들기

- 예전에는 예외 메시지를 단순하게 적는 경우가 많았지만, 예외 메시지가 불분명하면 AI는 여러 번 수정을 하게 된다.
- AI가 문제를 빠르게 식별할 수 있도록 구체적인 에러 메시지를 정의하도록 한다.
- assert에도 구체적으로 어떤 상황에서 문제가 발생하는지 적는다. (단순히 타입 확인용도는 생략하는 게 나음)
- 이를 이용하여 AI가 테스트 코드를 실행할 때 에러 메시지를 피드백으로 삼고 테스트 코드를 동작시킬 때 알아서 수정 하게 만든다. (셀프 피드백을 통해 코드의 완성도를 높인다.)

## 런타임 검증 사용하기

런타임 검증을 위해서 테스트 코드를 반드시 작성하도록 하자.

### 런타임 검증 최대화 하기

#### 예전

유저 생성 : 에전에는 개발 공수 시간 관계상 별도로 기능을 다 만들었다. 예를 들어 유저 정보 변경 이력이라고 하면 A 화면의 변경 갱신 그대로 B화면의 변경 이력은 갱신되지 않는 등의 문제가 발생하였다.

#### 지금

코드를 짜는 비용이 줄어들었다. 이를 적극 활용하여 AI가 실수를 덜 하도록 통제를 한다

#### 사례

모든 것이 유저 생성 서비스를 거쳐서 생성되게 한다. 유저 생성 서비스는 각종 도메인의 제약이 들어가 있는, DTO, ENUM, VO, 에러 메시지, assert 등이 들어간 잘못된 데이터가 들어가면 AI에게 바로 피드백을 할 수 있는 기능이다.

- 테스트 데이터를 만들 때 => 유저 생성 및 변경 서비스
- 유저 정보 편집 화면 => 유저 생성 및 변경 서비스
- 유저 정보 변경 csv => 유저 생성 및 변경 서비스
- 유저 계좌 정보 변경 => 유저 생성 및 변경 서비스

## 테스트 작성

### AI의 특성

AI는 일반적인 지식에 강하다. 허지만 회사의 코드는 비즈니스의 고유한 맥락 지식이 있기 때문에 틀릴 가능성이 높다.

### First, 유저/운용자의 입장을 생각하기

- 유저가 어떻게 조작을 하는가? 운용자가 어떻게 조작을 하는가?를 먼저 고려하여 테스트를 작성한다. (비즈니스 드리븐 디자인 BDD 관점을 테스트에 녹여 낸다.)
- 비즈니스 맥락에 의존하는 것으로 ai는 이 부분에 대해서 로직을 올바르게 판단하기 어려울 수 있다.
- 이 부분은 AI 가 틀릴 수 있기 때문에 사용 자의 직접적인 정의 사용자의 판단이 반드시 개입 해야 되는 부분이다.
- 테스트를 생성할 때 가장 먼저 생각해야 하는 부분이다.

### Second, 유저 인터페이스로 검증 불가능한 부분 대응

유저 인터페이스로 제공되지 않는 부분은 검증 하기 어려운 이 부분은 개발자가 해결해야 한다. 이것을 개발자의 판단에 따라 필요 하다고 생각하면 추가 지금은 필요 하지 않다면 생략을 해도 된다.

### Third, 엣지 케이스 대응

이 부분은 얘기치 않은 입력 또는 자주 사용 되지 않는 값에 대한 검증을 추가 하는 부분이다. 가능하면 테스트를 작성하면 좋지만. 개발자가 판단 하기 어려울 수도 있기 때문에 판단 할 수 있다면 추가 판단 할 수 없다면 생략. 나중에 문제가 발생 했을 때 추가 하도록 한다.

## TopDown 전략

- 바로 바이브로 코딩을 하지 않고, 디스커션, 스펙, 테스크로 작업을 나눈다.
- 이렇게 개발하면, 개발의 문서가 남는다. 개발 문서가 남으면 이 정보는 AI가 나중에 판단을 할 때 참고할 수 있게 되기 때문에 AI의 판단력이 좋아진다.

### 디스커션

현재 프로젝트의 구조를 분석하고, 새로운 과제를 프로젝트에 어떻게 적용할지 논의하는 과정이다. 이는 AI와의 상담과 조사 등을 통해서 작성하며, 개발자는 이 디스커션의 타당성을 검사한다.

### 스펙

수정 대상이 되는 부분을 모두 나열하고, 어떻게 수정하고, 어떤 전략으로 코드를 수정해 나갈 것인지를 정의한다. 최종적으로는 작업할 항목을 나열하고, 작업이 완료되면 완료된 테스크를 처리하도록 한다.

### 테스크

각 작업 항목에 대한 구체적인 테스크를 정의하는 부분이다. 이 지침서를 AI에게 전달하여 코드를 작성하게 한다.

### 테스트 스펙

스펙이 티켓의 작업 정의에 대한 것이었다면, 테스트 스펙은 티켓의 작업을 어떻게 테스트 할 것인지를 정의하는 곳이다. 앞서 테스트에 대한 설명을 했을 때, 유저/운용자의 관점에서의 동작을 정의하고, 개발자가 필요하다고 생각되는 영역의 테스트를 보강하는 정의를 하는 곳이다.

### 테스트 테스크

테스트 스펙의 각 항목을 구현하기 위해 정의하는 곳이다.

## 프로젝트에 정보를 주입하라

### 계약에 의한 코딩을 이용

- 특정 모듈에서 이용할 수 있는 값의 범위를 정한다. 다루는 범위를 AI에게 알려 준다.
- assert를 이용하면 AI가 테스트 코드를 실행할 때 빠른 피드백을 받아 코드를 올바르게 생성하게 유도할 수 있다.
- 예전에는 phpdoc을 작성할 때 언어 문법이 아니기 때문에 코드 자동완성이 없어 실수가 많았고, 이렇게 까지 타입 정의를 빡세게 사용하지는 않았으며, 타이핑에 시간이 들어서 작성하지 않았지만, AI를 사용하는 지금은 정확하게 작성할 수 있기 때문에 적극적으로 사용하게 한다.
- 이런 주석이 코드의 생성량을 늘리게 하지만, AI에게 가이드라인을 제공하여 코드의 완성도를 높일 수 있다.

#### phpstan에서의 지원

```
positive-int
negative-int
non-positive-int
non-negative-int
non-zero-int
int<0, 100>
int<min, 100>
int<50, max>
```

#### assert로 값의 범위 표현하기

```
// positive-int
assert(is_int($x) && $x > 0);

// negative-int
assert(is_int($x) && $x < 0);

// non-positive-int
assert(is_int($x) && $x <= 0);

// non-negative-int
assert(is_int($x) && $x >= 0);

// non-zero-int
assert(is_int($x) && $x !== 0);

// int<0, 100>
assert(is_int($x) && $x > 0 && $x < 100);

// int<min, 100>
assert(is_int($x) && $x >= $min && $x < 100);

// int<50, max>
assert(is_int($x) && $x > 50 && $x <= $max);
```

### faker를 사용하지말고 직접 테스트 데이터를 생성하라

AI롤 테스트 데이터를 만들기 위한 더미의 생성도 쉬워졌다. faker 라이브러리로 코드를 생성하는 것 보다, 모듈의 각종 제약사항을 통과할 수 있는 테이터를 작성할 수 있도록 한다.

### 도메인 주도 설계하기

도메인 주도 설계의 핵심은 도메인 로직을 얼마나 정확하게 코드에 적용하느냐에 따라 달렸다. 정확한 용어로 변수를 생성하여 AI가 개념을 혼동하지 않고 명확하게 구분할 수 있도록 하자.

## 증명 처럼 쌓아 올리는 코딩을 하라

외부 입출력 특히 디비나 컨트롤러 쪽에서 데이터를 받을 때 철저하게 통제하고 타입과 범위를 엄격하게 받도록한다. 그러면 이들을 조합하여 중간의 로직을 쌓아 올릴 수 있다.
