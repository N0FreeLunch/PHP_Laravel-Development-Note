# 공학적 시각 결여

하나의 프로젝트를 담당하는 인원이 점점 줄어들고 있다.

사람을 더 채용하기 보다는 기존 인원의 프로젝트 변경 등이 논의되는 시점이다.

프로젝트에 다른 사람이 담당하던 부분을 보게 되면서 이전의 코드 리뷰 정도로는 잡지 못했던 버그들을 마주하게 된다.

## 마주하는 문제

예를 들어 어떤 정보가 삭제 상태 (소프트 딜리트)인데 API 쪽에서는 이를 고려하지 않고 삭제 처리된 정보를 보여준다거나, API 전송시 실수가 있어서 무언가를 잘못 입력했을 때의 실수나 테스트 데이터가 다른 고객이 열람할 수 있는 로그에 포함되게 만들게 한다거나 로직을 만들 때 큰 주의를 기울이지 않고 작성하는 경우가 있다.

무언가를 개선하거나 수정을 해야 하는데, 원래 원인과 대상으로 지정되었던 것과는 다른 명백히 이건 잘못되었다고 판단되는 부분을 처리하게 되는 일이 생긴다. 어 이상한데, 왜 이렇게 만들었지? 그러니까 대략적인 리뷰, 대략적인 QA 만으로 잡지 못한 미묘한 로직의 다름으로 개선하지 않으면 안 될 것 같은 코드들이 검출된다.

문제는 API 관련이면, 한 번 정해진 사양이라면 수정하기가 쉽지 않고 고객이 사용중이라면 고객에게 연락을 해서 변경 사항이 있다는 것을 알려야 하고, 고객의 허락이 있을 때까지는 잘못된 사양을 함부로 변경하기 곤란해 질 수 있는 것이다. 삭제 데이터 계속 확인하고 있어서 바꾸면 안 되요. 그럼 저희 쪽에 공수가 들어갑니다. 이런 상황이 되었을 때, 잘못된 것을 고치기가 너무 까다로워지는 문제가 발생한다.

뭔가 하나를 수정하려고, 간단한 기능을 추가하려고 봤는데, 이전 로직이 잘못되어 있고, 단순히 동작을 안한다가 아니라 잘못 동작을 하기 때문에 이 로직을 기반으로 무언가를 더 쌓을 때 치명적이거나 보안상의 문제가 일어날 수 있는 부분이기 때문에 해당 부분을 수정하고 확인하고 테스트 하는 과정이 합쳐지면서 기대 일정을 훌쩍 넘겨 버릴 수 있는 사건이 일어난다.

## 엄격한 코드리뷰

애초에 엄격하게 코드 리뷰를 하면 되지만, 코드 리뷰가 엄격해질 수록 많은 시간을 소모한다. 그래서 눈에 보이는 쉬운 포인트를 지적하면서, 나중에는 이런 부분도 소프트웨어를 개발하면서 스스로 고려해서 잘 만들어 달라는 사람을 키우는 방식이 코드 리뷰이다. 모든 것을 리뷰어가 검증을 하는 게 아니라, 서로의 실력을 발전 시켜서 안전하고 견고하고 지속성있는 소프트웨어를 만들어나갈 수 있는 사람이 서로 되자는 것이다.

모든 것을 검증할 수는 없기 때문에 그리고 모든 것을 검증하면 상대방의 기분이 상할 수 있기 때문에, 적당히 서로의 관점을 공유하고 지적하고 성장함으로써 스스로 좋은 제품이 될 수 있도록 기여와 공헌을 할 수 있는 개발자로 성장해 주세요라는 것이 코드리뷰라고 생각한다.

## 공학적 관점 결여

오랫동안 서로 코드 리뷰를 하고 발전을 시켜서 제품에 더 좋은 기여를 하겠지만, 기본적인 위험성에 대해 너무 간과하는 케이스가 많다. 코드를 작성하는 능력의 문제가 아니라, 제품을 디자인하는 공학적인 관점 자체의 결여이다.

### 너의 잘하는 부분 인정해

일정 중요하지, 새로운 것 좋은 것 적용 중요하지, 코드 빠르게 만드는 것 중요하지, 집중하는 것 중요하지 그래 잘하는 것 좋아.

하지만 공학적 디자인 감각 결여는 정말 심각하고 위험하다. 결국 다른 사람이 봤을 때 결점이 있지만 방치하고 넘어 갈 수 있는 것을 만드느냐, 다른 사람이 도저히 간과할 수 없는 결점을 해결하지 않으면 안 되는 것을 만드느냐의 차이이기 때문이다.

코딩 테스트, 적성 검사 보다 먼저 공학적 디자인 능력을 판단해야 한다. 이게 없으면 결국 남이 해 줘야 한다. 하나의 결과물을 남이 다 뜯어 봐야 하고 검증해야 한다.

그럼 빨리 만들면 뭐해? 새로운 걸 적용해서 뭐해? 일정 맞추면 뭐해? 다른 사람이 너 대신 시간을 쓸 수 밖에 없는데, 공학적 디자인 시각 결여는 반드시 걸러야 하는 채용의 프로세스라고 본다.
