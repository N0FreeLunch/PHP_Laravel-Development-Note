# AI 시대 문제 해결

AI 등장 이전의 소프트웨어는 인간에 의한 결정론적 해결 방법을 주로 택했다.

그런데 AI가 등장하고 엄청난 양의 코드를 쏟아내고, 사람보다 문제를 설명을 더 잘하고, 사람보다 문제를 더 잘푸는 시대가 왔다.

이제 인간이 이를 통제할 수 없을 정도가 되었다. 소프트웨어 코드가 잘 변경되었다는 것을 어떻게 판단할 것인가?

예전처럼 더 이상 결정론적인 관점에서 모든 코드를 확인하는 것은 더 이상 미래가 없다.

## 다양한 문제 풀이 방법

문제를 푸는 방법은 다양해졌다. 결정론적 방법을 사용할 수 있다. 여전히 중요하다. 하지만 AI의 방대한 코드를 인간이 관리 할 수는 없다.

최근엔 굉장히 레거시한 코드가 있었는데 속도가 너무 느려서 리펙토링을 해야 했다. 오랜시간 수정을 했지만 결국에는 수정을 할 수 없었던 이유가 변경의 영향력을 평가하기라 너무 어려웠고, 코드를 해석하고 그와 동일한 사양의 새로운 코드를 작성하기 어려운 문제가 있었다.

개발 서버의 데이터를 이용해서 블랙박스인 AI를 통한 코드의 전후 결과가 완전히 일치하는 확인하는 방법의 개발을 통해서 시간을 단축시킬 수 있었다. 이걸 허용해야 하는지에 대해서는 미심쩍은 부분이 있지만, 어쨌든 기획이나 영업이 만족할 수 있는 시간 안에 해결할 수 있었다.

하나 하나 무슨 의미인지 로직을 파악하면서 그 로직을 어떻게 개선할지를 사람이 생각했다면 몇 주는 걸릴 것을 일주일에 해결할 수 있었다.

또 고전파의 테스트를 작성하기 위해서 데이터를 세팅하는데, 그냥 문자열로 예전에는 코드를 작성했다. 그러나 AI에게 문자열을 적고 테스트를 맡길 수 있을까? 공통의 서비스를 만들었다. 유저의 정보를 생성하는 기존 기능을 토대로 모듈을 만들고, 실제 애플리케이션의 유저 생성 동작도 유저 생성 서비스를 거치게 하고, 테스트 코드를 만들 때도 유저 생성 서비스를 거치게 하여 런타임의 검증을 통해서 생성하도록 했다.

이제는 블랙박스를 인정해야하지 않을까? 사람이 하던 시기에도 실수가 있었는데, 블랙박스를 어느 정도 허용해도 되지 않을까? 대신 어떻게 안전하게 블랙 박스의 안전성을 입증할 것인가?라는 생각을 해 보는 것도 하나의 방법일 수 있다.

AI가 생성한 로직이 블랙박스이지만, 이정도면 안전하다고 여기는 로직을 통과했어 그럼 써도 괜찮지 않을까? AI가 생성한 로직이 블랙박스이지만, 여러 데이터로 테스트를 해 보니 전후 데이터가 똑같은데 한 번 써도 되지 않을까?

이런 생각을 할 수 있다.

## 테스트

AI를 사용하면 아주 쉽게 다양한 데이터를 생성할 수 있다. 굳이 fake라는 도구를 쓰지 않아도 왠만한 케이스를 검증할 수 있는 데이터 셋을 만들 수 있다.

목킹을 적극 사용하는 런던파의 테스트는 그냥 통과만 하는 코드를 짤 수 있다. AI의 겉보기식 테스트 통과만을 생각할 것이 아니라 실제 런타임의 동작으로 올바른 데이터가 생성되었는지, 올바른 반환값이 이뤄졌는지를 확인하는 고전파의 테스트를 적극 활용해야 한다.

그리고 고전파의 테스트를 위해서 각종 도메인을 의미하는 애플리케이션의 각종 검증 로직으로 테스트 데이터도 필터링을 하는 방식을 사용해서 좀 더 강력한 안전장치를 만든다.

예전에는 코드가 가능한 적은 것이 유리했다면, 요즘에는 그냥 저렴하게 코드를 생성을 하되, 안전하게 코드를 늘리는 방법도 생각해 볼 수 있을 것 같다.

## 다면적 다층적 리뷰

사람이 코드 리뷰를 하는 시대는 지났다. AI에게 맡겨서 코드를 작성하게 하는 편이 낫다. 요즘 코드 리뷰는 AI가 프로젝트에 설정된 각종 룰을 따르는지 확인을 하고 코드 리뷰까지 해 준다.

프로젝트에 각종 코딩 스타일 및 룰을 잘 설정했다면 더 이상 코드 스타일에 대해 사람이 굳이 따질 필요가 없다. 물론, AGENTS.md, skills에 룰이 정해지기 까지는 논쟁이 있을 수 있고, 룰에 대한 이의를 제기 할 수도 있지만, 일단 정해진 룰이라면 AI는 그 기준을 잘 판단해 줄 것이다.

하지만, 인간의 리뷰는 필요하다. 무엇을 리뷰 할 것인가? PHP라면 phpstan 등으로 뽑은 시그니처의 관계도를 통해서 리뷰를 할 수 있을 것이다. 고 언어면 AST로 얻은 시그니처 관계도를 활용할 수 있다.

그리고 어떻게 AI라는 블랙 박스를 다뤄서 문제를 해결했고, 그 안정성을 검증하는 방법론이 타당한지를 확인하면 된다.

그리고 AI와 만든 사양서, 최종적 결과물에 대한 내용을 AI에게 지시하여 업데이트 한 사양서의 타당성을 검증하는 방법도 쓸 수 있다.

등을 생각할 수 있다. 예전과는 다른 방식을 사용해야 하며 블랙 박스를 어떻게 다루고 통제할 수 있을까를 리뷰의 새로운 기준으로 삼아야 한다.
