# AI 시대 문제 해결

AI 등장 이전의 소프트웨어는 인간에 의한 결정론적 해결 방법을 주로 택했다.

그런데 AI가 등장하고 엄청난 양의 코드를 쏟아내고, 사람보다 문제를 설명을 더 잘하고, 사람보다 문제를 더 잘푸는 시대가 왔다.

이제 인간이 이를 통제할 수 없을 정도가 되었다. 소프트웨어 코드가 잘 변경되었다는 것을 어떻게 판단할 것인가?

예전처럼 더 이상 결정론적인 관점에서 모든 코드를 확인하는 것은 더 이상 미래가 없다.

## 다양한 문제 풀이 방법

문제를 푸는 방법은 다양해졌다. 결정론적 방법을 사용할 수 있다. 여전히 중요하다. 하지만 AI의 방대한 코드를 인간이 관리 할 수는 없다.

### 사례 1

최근엔 굉장히 레거시한 코드가 있었는데 속도가 너무 느려서 리펙토링을 해야 했다. 오랜시간 수정을 했지만 여전히 남겨 두었던 이유가 변경의 영향력을 평가하기가 너무 어려웠고, 코드를 해석하고 그와 동일한 잘 리펙토링된 새로운 코드를 작성하기 어려운 문제가 있었다.

개발 서버의 데이터를 이용해서 블랙박스인 AI를 통한 코드의 전후 결과가 완전히 일치하는지 확인하는 방법의 개발을 통해서 시간을 단축시킬 수 있었다. 이걸 허용해야 하는지에 대해서는 미심쩍은 부분이 있지만, 어쨌든 기획이나 영업이 만족할 수 있는 시간 안에 해결할 수 있었다.

하나 하나 무슨 의미인지 로직을 파악하면서 그 로직을 어떻게 개선할지를 사람이 생각했다면 몇 주는 걸릴 작업을 일주일에 해결할 수 있었다. 물론 언제든지 새로운 코드로 작성된 로직을 되돌리고 과거 로직으로 로직을 동작 시킬 수 있는 안전 장치를 마련해 둔 상태에서 진행을 하였다.

### 사례 2

고전파 테스트를 위해서는 실제로 사용될만한 데이터 세팅이 필수적인데, 예전에는 이를 사람이 하드코딩하는 경우가 많았다. 그러나 AI에게 문자열을 적고 테스트를 맡길 수 있을까? 공통의 서비스를 만들었다. 유저의 정보를 생성하는 기존 기능을 토대로 모듈을 만들고, 실제 애플리케이션의 유저 생성 동작도 유저 생성 서비스를 거치게 하고, 테스트 코드를 만들 때도 유저 생성 서비스를 거치게 하여 런타임의 검증을 통해서 생성하도록 했다.

각종 도메인의 제약 사항이 담긴 서비스를 이용해서 데이터의 생성을 담당하게 하였고 이릍 통해서 안전한 데이터를 생성할 수 있었다.

### 블랙 박스 로직 피할 수 없다면 즐겨라

이제는 블랙박스를 인정해야하지 않을까? 사람이 하던 시기에도 실수가 있었는데, 블랙박스를 어느 정도 허용해도 되지 않을까? 대신 어떻게 안전하게 블랙 박스의 안전성을 입증할 것인가?라는 생각을 해 보는 것도 하나의 방법일 수 있다.

AI가 생성한 로직이 블랙박스이지만, 이 정도면 안전하다고 여기는 로직을 통과했어 그럼 써도 괜찮지 않을까? AI가 생성한 로직이 블랙박스이지만, 여러 데이터로 테스트를 해 보니 전후 데이터가 똑같은데 한 번 써도 되지 않을까?

이런 생각을 할 수 있다. 단 AI가 생성한 모든 코드를 이해할 수는 없더라도, 이 정도면 나중에 수정할 수 있겠다는 감각의 정도로 코드 관리는 해야 한다. 어떤 문제가 발생했는데 갑자기 AI 서비스가 중단된다면 어떻게 해결할 것인가?

## 테스트

### 실무에서 진짜 사용되는 데이터 셋 만들기

AI를 사용하면 아주 쉽게 다양한 데이터를 생성할 수 있다. 굳이 faker라는 도구를 쓰지 않아도 웬만한 케이스를 검증할 수 있는 데이터 셋을 만들 수 있다.

faker 라이브러리를 이용한 테스트의 문제는 서비스에서 제공하는 폼 입력 케이스에 맞지 않는 데이터가 있을 수 있다는 점이다.

예전에는 일일이 테스트 하기에 적합한 케이스를 가진 데이터를 생성하기 어려웠기 때문에 faker를 썼으나, 요즘에는 이런 더미 데이터를 생성하기 무척 쉬워졌다.

생성한 더미 데이터를 고정해 두고 예외 사항이 발생하며 계속 추가하여 검증 데이터 셋을 확보하는 방식을 사용하는 것이다. 그리고 규칙이 있다면 조합을 하는 방식으로 그 수를 늘릴 수 있다. 예를 들어 사람의 성, 이름이 있다면 이를 조합해서 쑬 수 있을 것이다.

AI 시대에는 도메인 제약을 가진 각종 런타임 검증이 더 중요해진 시대이다. 모든 결정론적 로직을 확인 할 수 없는 시대라면 각종 제약사항을 통해서 가능한 안전한 데이터를 생성해야 한다. AI가 코드의 제약 사항을 보고 적절히 데이터 세트를 생성할 수 있으며 faker 라이브러리는 서비스의 제약 사항을 반영하지 않기 때문에 이제는 더 이상 적절하지 않을 수 있다.

### 애플리케이션의 동작을 직접 검증하기

목킹을 적극 사용하는 런던파의 테스트는 그냥 통과만 하는 코드를 짤 수 있다. AI의 겉보기식 테스트 통과만을 생각할 것이 아니라 실제 런타임의 동작으로 올바른 결과가 생성되었는지 확인하는 고전파의 테스트를 적극 활용하여 AI가 잘못된 테스트를 작성하지 않도록 제한을 가하는 것이 중요하다.

그리고 고전파의 테스트를 위해서 각종 도메인을 의미하는 애플리케이션의 각종 검증 로직으로 테스트 데이터를 필터링하여 로직을 동작시키기 전에 필요한 전제 데이터를 정확하게 생성하도록 한다.

동적 언어는 AI 시대에 상대적으로 불리한 입지에 있었다. 그러나 강력한 런타임 검증 방식을 적극 도입하는 것을 통해 전체 어플리케이션을 횡단하는 런타임 테스트가 상대적으로 불편한 컴파일 언어에 비해 이점을 가져갈 수 있다. 특히 PHP 등을 사용하면 적당한 타입과 모든 타입이 런타임에 결정하는 것과 assert를 통한 벨리데이션 등 강력한 런타임 검증 로직을 만들 수 있다는 장점이 있다.

### 통제된 방법론을 활용한 블랙박스 다루기

예전에는 잘 만들어진 라이브러리, 언어에 존재하는 각종 내장 함수, 외부 서비스 등을 잘 활용해서 코드를 적게 유지하는 것이 유리했다. 중첩된 for문을 작성하다가 실수를 하는 경우도 있었고, 어차피 좀 복잡한 개념을 만들거면 잘 알려진 함수를 사용하면, 읽는 입장에서 알아야 할 게 하나 더 늘어나지만 어차피 언어를 배우면서 결국에는 습득을 하게 될 지식이기 때문에 (너무 예외적인 것은 피하고) 코드를 복잡하게 짜는 것 보다 잘 추상화된 단위의 개념의 조합을 쓰는 것이 나았다.

하지만 AI 시대에는 코드 생성 비용이 저렴하므로, 도메인 제약을 위해 생성한 여러 검증 도구를 이용해서 강력한 런타임 검증을 거치는 방식으로 AI가 생성한 코드를 점진적으로 늘리는 방식의 접근법을 생각해 볼 수 있다.

하지만 코드는 사람이 이해하고, AI도 자기가 짠 복잡성을 해결하지 못할 수준의 코드까지 가지 않게 하기 위해서 무작적 코드를 늘리지 않게 하기 위해서 적절한 룰의 부여와, 코드를 생성할 때 코드 생성 지침과 개발자의 판단에 의한 전략을 작성하여 허용 가능한 범위에서 코드를 생성할 수 있도록 하는 것이 중요하다.

## 다면적 다층적 리뷰

### 인간 코드 리뷰 시대의 끝

사람이 코드 리뷰를 하는 시대는 지났다. AI에게 맡겨서 코드를 작성하게 하는 편이 낫다. 요즘 코드 리뷰는 AI가 프로젝트에 설정된 각종 룰을 따르는지 확인을 하고 코드 리뷰까지 해 준다.

프로젝트에 각종 코딩 스타일 및 룰을 잘 설정했다면 더 이상 코드 스타일에 대해 사람이 굳이 따질 필요가 없다. 물론, AGENTS.md, 에이전트 스킬의 룰이 정해지기 까지는 논쟁이 있을 수 있고, 룰에 대한 이의를 제기 할 수도 있지만, 일단 정해진 룰이라면 AI는 그 기준을 잘 판단해 줄 것이다.

#### 코드를 다 읽어야 할 때

모든 프로젝트는 점진적으로 AI 네이티브 환경을 만들려고 할 것이다. 이를 위해 AI의 출력 코드가 만족할만한 대상인지를 확인을 해야 한다. 요즘 AI를 쓰지만 생각보다 시간이 많이 걸리는 이유도 AI의 출력 결과물을 보고 에이전트 스킬이나 AGENTS.md 파일을 세팅하기 때문이다. 이 때문에 코드를 읽을 수 밖에 없다.

하지만 이러한 세팅이 어느 정도 완료 되었다고 한다면, 이제는 코드를 작성한 개인이 자신의 코드를 셀프로 리뷰를 진행하고, 타인인 동료는 모든 코드를 읽지 않고, 제출자가 찝어낸 논의 사항 및 좀 더 간략하지만 중요한 것을 확인할 수 있도록 해야 한다.

그리고 한 번의 실수가 서비스 전체게 너무나도 크리티컬한 영향을 주는 경우가 있을 수 있다. 특히 데이터베이스 스키마 변경, 마이그레이션은 데이터를 완전 소실 시킬 수도 있는 부분이다. 앞으로의 기업의 가치는 데이터가 되는데 데이터가 소실되면 돌이킬 수 없는 타격을 가한다. 그래서 이러한 부분은 인간의 직접적인 검수가 반드시 필요하다.

### 인간은 무엇을 리뷰 할 것인가?

#### 시그니처를 통한 확인

하지만, 인간의 리뷰는 필요하다. PHP라면 php 파서 등으로 얻은 AST의 시그니처를 통해서 리뷰를 할 수 있을 것이다.

#### 블랙 박스를 다루는 방법론 확인

그리고 어떻게 AI라는 블랙 박스를 다뤄서 문제를 해결했고, 그 안정성을 검증하는 방법론이 타당한지를 확인하면 된다.

#### 변경 사항이 초래하는 영향력 판단

얼마전 클로드 코드 개발자가 개발 환경의 편의를 위해 소스맵 기능을 추가했는데 노출되면서 비공개 코드인 클로드에서 어떤 기능이 구현되어 있는지 세부적으로 알 수 있게 되었다. 공개와 비공개의 원칙을 잘 지키고 있는가?

디비의 데이터 삭제 등의 크리티컬한 영역에 대한 충분한 판단 능력을 보유하고 있는가 등이 기준이 될 것이다.

#### 사양서의 최종 결과물을 검토

그리고 AI와 만든 사양서, 최종적 결과물에 대한 내용을 AI에게 지시하여 업데이트 한 사양서의 타당성을 검증하는 방법도 쓸 수 있다.

#### 새로운 기준이 필요하다.

예전과는 다른 방식을 사용해야 하며 블랙 박스를 어떻게 다루고 통제할 수 있을까를 리뷰의 새로운 기준으로 삼아야 한다.

## 앞으로 개발자의 공부 방향

개발자의 공부의 집중적인 탐구 방향은 크리티컬한 문제를 어떻게 다룰 것인지에 대한 부분을 집중 타격하는 공부가 되어야 한다. 프로그래밍 문법을 배우고, 코드를 직접 작성하는 것 보다 더 중요한 부분이다. 데이터 베이스의 변경 방향에 대한 이해, 공개 되어야 할 정보와 공개 되지 않아야 할 정보가 무엇인지 구분하여, AI의 작업 결과가 어떤 결과를 초래하는지 생각을 할 수 있어야 한다.
